```
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        BACKEND                               │
├─────────────────────────────────────────────────────────────┤
│  Supabase (PostgreSQL + Auth + Storage + RPC)               │
│  Row Level Security (RLS) para permisos                     │
│  Functions PostgreSQL (api_*)                               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      DEPLOYMENT                              │
├─────────────────────────────────────────────────────────────┤

└─────────────────────────────────────────────────────────────┘
```

### Decisiones Arquitectónicas Clave

| Decisión | Justificación | Beneficio |
|----------|---------------|-----------|
| Supabase vs Firebase | PostgreSQL relacional, RLS nativo, costos | Escalabilidad + Seguridad |
| PWA vs App nativa | Un solo codebase, offline-first | Reducción 60% tiempo |
| Zod + RHF | Validación type-safe, mejor DX | Menos bugs en forms |
| RPC Functions | Lógica en BD, menos APIs | Consistencia + Performance |

## 🎓 9. LECCIONES APRENDIDAS

### Lo que funcionó bien ✅

1. **IA para scaffolding rápido** - Estructuras base de componentes en segundos
2. **Iteración con prompts** - Refinamiento incremental más efectivo que planificación extensa
3. **Documentación inline** - Copilot genera comments y docs automáticamente
4. **Debugging asistido** - Identificación de errores más rápida

### Desafíos encontrados ⚠️

1. **ENUMs PostgreSQL** - Causaron errores en funciones JSON (mitigado con VARCHAR)
2. **Contexto en conversaciones largas** - Copilot "olvidaba" código previo
3. **Consistencia de tildes** - Discrepancias entre capas (solucionado con normalización)
4. **Diseño inicial no escalable** - Arquitectura scouts-céntrica requirió refactorización a personas+roles (abr 2026)

### Recomendaciones para futuros proyectos

1. Usar **VARCHAR con CHECK** en lugar de ENUMs en PostgreSQL
2. Mantener **sesiones de trabajo cortas** (2-3 horas máximo)
3. **Documentar decisiones** inmediatamente (archivos IMPLEMENTACION_*.md)
4. **Commits frecuentes** para tener checkpoints de restauración
5. **Diseñar arquitectura escalable desde el inicio** - considerar múltiples roles/estados desde día 1
